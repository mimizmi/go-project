# 医院异构系统实时数据中台设计与实现——多数据库CDC采集、Exactly-Once投递、增量位点管理

## 选题依据
1、拟研究的方向和主要内容

随着医院信息化系统不断增多，院内 HIS、EMR、LIS、PACS 等业务系统往往由不同厂商建设，数据库类型与数据标准不统一，导致数据孤岛与共享困难，跨系统统计分析依赖批处理 ETL/数据仓库，存在明显时延，难以支撑急诊抢救、联合会诊、实时运营监控等“秒级可用”的业务诉求。基于此，本课题拟面向医院多源异构数据库，研究并实现一套实时数据中台原型系统，重点围绕“增量采集—可靠传输—一致性投递—容错恢复—模式演进”开展。
主要研究内容包括：
（1）构建面向多数据库的CDC适配框架，基于事务日志实现非侵入式增量采集与统一事件封装；
（2）设计基于 Kafka 的实时消息总线与事务投递机制，实现端到端 Exactly-Once语义；
（3）研究多数据源增量位点一致提交策略与断点续传容错流程；

2、研究意义

理论意义上，本研究面向医院复杂异构环境，将 CDC、流式处理一致性语义、位点容错等关键问题系统化建模与组合验证，为实时数据中台的可复用架构与工程方法提供参考。实践意义上，原型系统可在不改造既有业务系统的前提下，实现跨库数据的低延迟汇聚与共享，提升临床决策与运营管理的时效性；同时通过一致性投递与可追溯位点管理，降低“重复/遗漏”带来的数据风险，为医院后续数据治理与数据服务化奠定底座。

3、国内外相关研究情况

国内外实时数据集成常采用 CDC+消息队列+流处理的架构路线:CDC 工具（如 Debezium、Flink CDC、商业 GoldenGate）可从日志侧捕获增量变更并推送至消息系统；Kafka等平台通过幂等与事务能力支持“处理与提交的原子性”，成为实现 Exactly-Once的关键基础；在数据服务与分析侧，“湖仓一体”被广泛讨论，用于兼顾历史数据沉淀与实时增量更新。与此同时，医疗数据平台建设强调多源异构汇聚、标准化治理与安全合规，但在医院“多数据库并存 + 老旧系统适配 + 全链路 Exactly-Once”场景下，仍存在工程落地复杂、位点一致性难、跨系统重复投递难彻底避免等问题，亟需面向场景的系统性设计与原型验证。

 ## 研究方案

 1、研究方法及技术路线

（1）研究方法：采用“设计科学研究(Design Science)+原型系统实现+对比实验评估”的方法。先从医院异构系统的数据流转与实时需求出发，抽象关键技术问题；再给出可实现的架构方案并完成原型；最后通过功能正确性、容错一致性与性能指标（延迟、吞吐、资源占用）实验验证其有效性。
（2）技术路线：

数据源侧(CDC 采集)：选取 2–3 类常见医院数据库(如 MySQL、SQL Server 等)作为实验源。CDC 组件基于事务日志解析（binlog/redo/transaction log 等）捕获 INSERT/UPDATE/DELETE 事件，统一封装为标准化“变更事件”（包含表标识、主键、前后镜像、事务边界、时间戳、位点信息等），并支持过滤/脱敏等预处理。
消息队列侧（Kafka 事务投递）：以 Kafka 作为数据总线，将不同源库/表映射到主题与分区以保障顺序性。Producer 端启用幂等与事务，在“发送数据—提交位点”之间建立原子绑定：只有当事务提交成功，才对外可见并确认位点；若失败则回滚并重试，避免重复影响下游。Kafka Streams 等框架在 Exactly-Once 方面提供成熟语义支撑，可作为端到端验证的参照实现。
处理服务侧（消费与落地）：下游可实现两类服务：（a）实时汇聚服务：把多源变更写入统一 ODS/实时数据仓库，形成“近实时一体化视图”；（b）数据服务接口：对外提供查询/订阅 API，实现 Data-as-a-Service。消费端同样使用事务/幂等写入或检查点机制，确保一次生效。
位点管理与容错：设计“全局一致位点提交策略”，以源库位点+Kafka offset+事务状态为核心元数据，落地到可靠存储；故障恢复时先判定最后一次成功事务边界，再从对应位点继续，实现断点续传。
2、课题拟解决的关键问题

（1）异构数据库 CDC 适配：不同数据库日志格式与事务语义差异导致统一采集与事件标准化困难；
（2）端到端 Exactly-Once：采集、传输、消费全链路一致性，需解决事务边界、重试与幂等写入协同；
（3）多源位点一致提交：多源并发下的位点推进与提交原子性，避免“提交了位点但数据未生效/数据生效但位点未提交”；
（4）容错恢复与数据不重不漏：进程/节点故障、网络抖动、Kafka 重平衡等场景下的恢复策略；
（5）安全与合规：医疗数据敏感，需在采集/传输/落地环节具备脱敏、权限控制与审计能力。

3、实验方案

（1）实验环境：

源数据库：MySQL、SQL Server（可扩展）；
消息队列：Kafka 集群（多 broker，多分区，多副本）；
处理与落地：消费服务（自研），目标库
监控：Prometheus+Grafana（或等价方案）采集端到端指标。
（2）数据与场景：构造模拟医院核心业务表（患者、就诊、检验、医嘱等）与典型事务（多表更新、批量写入、并发更新、回滚事务）。
（3）验证指标与方法：
正确性：校验目标库与源库在任意时刻的一致性（抽样比对+校验和）；
Exactly-Once：对比 At-Least-Once 与 EOS 模式下重复率/漏数；
容错：故障注入（重启 CDC、断网、broker 宕机、消费者重平衡），验证恢复后是否不重不漏；
性能：不同分区数、批量参数下的吞吐与端到端延迟（P50/P95/P99）；

## 文献综述内容
一、引言

医院信息系统长期呈现“多系统并行建设、标准不一、数据库异构”的客观状态，临床业务与运营管理对数据的依赖不断增强，但传统以 T+1 为主的批量 ETL 与数据仓库链路难以满足实时分析与联动决策。近年来，随着事件流与流处理生态成熟，“CDC+消息队列+流式计算/服务”逐渐成为实时数据平台主流路线。医疗领域在建设大数据平台时普遍强调多源异构汇聚、数据治理与隐私保护，但医疗数据链路的实时性与一致性要求更高：一旦存在重复或遗漏，会直接影响指标口径与临床判断。因此，有必要围绕 CDC 采集、Exactly-Once 语义、位点容错等关键技术，形成面向医院场景的系统化方案与原型验证。

二、国内外研究现状

1）医疗多源异构数据汇聚与平台化建设

医疗大数据平台研究普遍将“数据汇聚—治理—共享”作为主线，强调统一标准、数据目录、质量控制与安全机制。以多中心医疗大数据平台为例，研究指出医疗数据天然具备多源异构、标准多样与隐私要求高等特征，平台架构常采用组件化/模块化方式以增强扩展性，并通过可视化治理工具提升一致性与可追溯性。区域级平台研究则更强调跨机构的数据目录与标准规范建设，以及安全共享机制，推动跨系统互联互通与协同服务。这些工作为医疗数据平台的治理与共享提供了实践基础，但对“跨异构库的持续增量同步、低延迟供给与全链路一次生效”的讨论相对不足。

2）CDC（变更数据捕获）与实时集成路线

CDC 通过解析数据库日志或变更接口捕获增量事件，相比全量抽取可显著降低源库压力并缩短时延。国内关于日志型 CDC 的研究多聚焦在多源异构变更的捕获与更新机制设计，强调在多源并发下对变更事件进行统一建模、顺序控制与可靠更新，以提升同步正确性与效率。在工程生态方面，Flink CDC、Debezium 等逐渐从“单库单表同步”向“多表同步+结构变化处理+端到端管道化”演进，说明 CDC 正在成为实时数据中台的关键底座能力。但在医院场景中，源系统往往存在老旧版本、定制字段、复杂事务等特点，通用工具仍需结合业务约束进行适配与治理。

3）Exactly-Once 语义与端到端一致性

流式系统常见语义包括 At-Most-Once、At-Least-Once 与 Exactly-Once。医疗数据场景通常更关注“一次生效”，因为重复会造成指标膨胀、遗漏会造成缺口。Kafka 在幂等与事务能力基础上，使生产与消费侧可以将“写入数据”和“提交位点/状态”绑定为原子操作，为端到端 Exactly-Once 提供了关键支撑；Kafka Streams 等也在官方文档中强调其对容错与一致性处理的支持。但将 Exactly-Once 从消息层扩展到“CDC 采集—传输—落地存储”全链路，仍面临事务边界对齐、跨系统幂等键设计、重试与回放控制等工程挑战。

4）位点管理、容错恢复与“不断点”的持续运行

实时数据管道需要长周期稳定运行。位点管理的目标不仅是“记住同步到哪里”，更要在故障与重启后做到“不重不漏”。在多源 CDC 场景中，位点往往是“源库日志位置+Kafka offset+事务状态”的组合，任何一个环节不一致都可能导致重复或遗漏。因此，研究与工程实践通常强调：位点提交策略必须与事务提交严格绑定；恢复时必须以最后一次成功事务边界作为重放起点；并且要对异常（网络抖动、分区迁移、消费者重平衡）设计可验证的恢复流程。

三、可供参考的系统流程设计（面向本课题的归纳）

综合上述研究，本课题原型系统流程可归纳为：
（1）源库 CDC 采集：日志解析→统一事件封装→（可选）脱敏/过滤；
（2）消息总线：按库/表/业务域划分主题与分区→事务发送→持久化与复制；
（3）消费处理：订阅事件→幂等/事务落地→对外数据服务；
（4）治理与运维：位点一致提交→故障检测与恢复→监控告警。

四、结束语

总体来看，医疗领域的数据平台研究已形成较成熟的“汇聚治理与共享”框架，但对“异构库增量采集+端到端 Exactly-Once+位点容错”一体化落地仍存在缺口。本课题拟以医院典型异构库为对象，构建可运行的原型链路，通过严格的故障注入与一致性验证，形成可复用的技术路线与工程实践参考。
